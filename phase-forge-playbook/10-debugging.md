# Отладка: персистентные сессии с DEBUG.md

Отладка в Phase Forge — не "почини и забудь", а структурированный процесс
с персистентным состоянием, которое переживает очистку контекста.

---

## Принцип

DEBUG.md — это **мозг отладки**. Агент может возобновить расследование
после `/clear` без потери прогресса:
1. Прочитать frontmatter → знать статус.
2. Прочитать Current Focus → знать что делалось.
3. Прочитать Eliminated → знать что НЕ пробовать снова.
4. Прочитать Evidence → знать что уже найдено.
5. Продолжить с next_action.

---

## Жизненный цикл

### 1. Gathering (сбор симптомов)
```markdown
## Symptoms
expected: Пользователь видит логотип чётко
actual: Экран тёмный, логотип не виден
errors: Нет ошибок в консоли
reproduction: Открыть /auth страницу
started: После последнего деплоя
```

### 2. Investigating (расследование)
```markdown
## Current Focus
hypothesis: CSS переменная --bg-color переопределяется
test: Проверить computed styles в DevTools
expecting: Если --bg-color dark → подтверждает гипотезу
next_action: Найти где переопределяется переменная

## Evidence
- timestamp: 2025-01-20T10:15
  checked: app/static/style.css
  found: --bg-color: #1a1a1a (тёмный)
  implication: Фон задаётся явно тёмным цветом

## Eliminated
- hypothesis: Логотип не загружается
  evidence: Network tab показывает 200 для logo.svg
  timestamp: 2025-01-20T10:10
```

### 3. Fixing (исправление)
```markdown
## Resolution
root_cause: CSS переменная --bg-color установлена в #1a1a1a вместо #ffffff
fix: Изменить --bg-color на #ffffff в style.css
verification: Открыть /auth → логотип виден чётко
files_changed: [app/static/style.css]
```

### 4. Resolved (решено)
Файл перемещается в `.planning/debug/resolved/`.

---

## Правила секций

| Секция | Режим записи | Назначение |
|--------|-------------|------------|
| Frontmatter | OVERWRITE | Статус, триггер, timestamps |
| Current Focus | OVERWRITE | Что агент делает прямо сейчас |
| Symptoms | IMMUTABLE | Описание проблемы (фиксируется один раз) |
| Eliminated | APPEND only | Отвергнутые гипотезы (никогда не удалять) |
| Evidence | APPEND only | Найденные факты |
| Resolution | OVERWRITE | Корневая причина и фикс |

**Критически важно:** Eliminated секция **никогда** не удаляется.
Это предотвращает повторное расследование тупиковых направлений после `/clear`.

## Параллельная диагностика

При нескольких проблемах (например, после UAT) — запускай
**параллельных** debug-агентов:

```
UAT нашёл 3 проблемы
    │
    ├── Debug Agent 1: issue-auth-dark  (свежий контекст)
    ├── Debug Agent 2: issue-form-error (свежий контекст)
    └── Debug Agent 3: issue-slow-load  (свежий контекст)
    │
    ▼
Результаты: 3 root causes → план исправлений
```

Каждый агент работает изолированно со своим DEBUG.md.

---

## Правила отладки

1. **Минимальный фикс:** 1 баг = 1 фикс. Без "починить заодно".
2. **Регрессионный тест:** если возможно — добавить тест, который воспроизводит баг.
3. **Не ходить кругами:** если > 10 записей в Evidence — проверить Eliminated.
4. **Эскалация:** если 3 гипотезы отвергнуты подряд → остановиться, спросить.
5. **Фиксация:** все значимые находки → в Evidence, все тупики → в Eliminated.

---

## Quick Debug (без полной сессии)

Для простых багов можно обойтись без персистентной сессии:

```
1. Воспроизвести.
2. Сформулировать 2-3 гипотезы.
3. Проверить самую вероятную.
4. Минимальный фикс.
5. Регрессионный тест.
```

Используй полную DEBUG.md сессию, когда:
- Баг сложный или неочевидный.
- Несколько сессий отладки.
- Нужно не потерять контекст расследования.
