# Планирование: формат PLAN.md

PLAN.md — атомарный план выполнения. Содержит задачи в XML-формате,
метаданные для волнового выполнения и критерии верификации.

---

## Структура PLAN.md

### Frontmatter (YAML)

```yaml
---
phase: 3
plan: 2
title: "User Profile API"
wave: 1
depends_on: []
files_modified:
  - src/api/profiles.py
  - src/models/user.py
  - tests/test_profiles.py
autonomous: true
---
```

| Поле | Назначение |
|------|------------|
| `phase` | Номер фазы |
| `plan` | Номер плана внутри фазы |
| `title` | Краткое название |
| `wave` | Волна выполнения (1, 2, 3...) |
| `depends_on` | Планы, от которых зависит: `["03-01"]` |
| `files_modified` | Какие файлы будут затронуты |
| `autonomous` | `true` если может выполняться без остановок |

### Задачи (XML-формат)

```xml
<task type="auto">
  <name>Создать модель User Profile</name>
  <files>src/models/user.py</files>
  <action>
    Добавить поля display_name (str, required), bio (str, max 500),
    avatar_url (str, nullable). Использовать SQLAlchemy declarative.
    Миграция Alembic.
  </action>
  <verify>pytest tests/test_models.py -x --tb=short</verify>
  <done>Модель создана, миграция применима, тест проходит</done>
</task>
```

### Типы задач

| Тип | Описание |
|-----|----------|
| `auto` | Выполняется автоматически, без остановок |
| `human-verify` | Требует ручной проверки результата |
| `human-action` | Требует действия человека (настройка сервиса, ввод ключей) |

### Секция must_haves

Критерии из Success Criteria фазы, которые **обязаны** быть выполнены:

```markdown
## must_haves

1. Пользователь может создать профиль с display_name
2. API возвращает профиль по ID
3. Невалидный запрос → 422 с описанием ошибки
```

Эти критерии проверяются верификатором **от целей назад к коду**
(goal-backward verification).

## Волновое распределение (Wave Assignment)

### Правила

1. **Волна 1:** планы без зависимостей → параллельное выполнение.
2. **Волна 2:** планы, зависящие от волны 1.
3. **Волна N:** планы, зависящие от волны N-1.

### Определение зависимостей

- **Зависимость по данным:** Plan B использует модель, созданную Plan A.
- **Зависимость по API:** Plan B вызывает endpoint, созданный Plan A.
- **Конфликт файлов:** оба плана модифицируют один файл → последовательно.

### Пример

```
Фаза 3: E-commerce Cart

Plan 03-01: Product Model + API      wave: 1, depends_on: []
Plan 03-02: Cart Model + API         wave: 1, depends_on: []
Plan 03-03: Cart-Product Integration  wave: 2, depends_on: ["03-01", "03-02"]
```

Планы 01 и 02 → параллельно (волна 1).
План 03 → ждёт завершения обоих (волна 2).

## TDD-планы

Когда фаза описывает чётко тестируемое поведение, используй TDD:

```yaml
---
type: tdd
---
```

### Цикл Red-Green-Refactor

1. **Red:** написать тест, который падает (целевое поведение).
2. **Green:** минимальная реализация, чтобы тест прошёл.
3. **Refactor:** улучшить код без изменения поведения.

### Коммиты для TDD

```
test(03-01): добавить тест профиля (red)
feat(03-01): реализовать модель профиля (green)
refactor(03-01): извлечь валидацию в отдельный метод
```

## Scope-контроль

### Размер плана
- 2–3 задачи на план (не больше).
- Если задач > 3 → разбей на два плана.
- Каждая задача должна помещаться в один контекст (~50% окна).

### Антипаттерны
- **"Сделать всё"** — один план на всю фазу. Разбивай.
- **Горизонтальные слои** — "все модели", "все API". Предпочитай вертикальные срезы.
- **Безразмерные задачи** — "реализовать фичу". Конкретизируй.
- **Пропуск verify** — каждая задача должна иметь `<verify>`.

### Хороший план (вертикальный срез)
```
Plan 03-01: User Registration (модель + API + тест)
Plan 03-02: Email Verification (сервис + API + тест)
```

### Плохой план (горизонтальные слои)
```
Plan 03-01: Все модели (User, Email, Session)
Plan 03-02: Все API endpoints
Plan 03-03: Все тесты
```

Вертикальные срезы параллелизируются лучше.
