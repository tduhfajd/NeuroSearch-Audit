# Agent Policy: правила и ограничения для ИИ-агента

Эти правила задают границы для агента. Включай ссылку на этот файл
в каждый промпт или через `AGENTS.md`.

---

## 1) Обязательный контекст

Перед началом работы агент **обязан** прочитать:
1. `AGENTS.md` — контракт агента и границы проекта.
2. `STATE.md` — текущая позиция и накопленный контекст.
3. Спецификацию текущей задачи (PLAN.md или spec).

Без этого контекста агент не должен писать код.

## 2) Разрешено

- Читать файлы, искать по репозиторию, объяснять код.
- Выполнять задачи строго по текущему PLAN.md.
- Делать **минимальный diff** в рамках одной задачи.
- Добавлять/обновлять тесты рядом с изменением.
- Запускать проверки (линтер/тесты/типизация) и чинить найденное.
- Обновлять артефакты: STATE.md, SUMMARY.md, CHANGELOG.md.
- Коммитить каждую задачу отдельно (атомарные коммиты).

## 3) Запрещено без явного подтверждения

- Выход за scope текущего плана/фазы.
- Массовые рефакторинги, переименования.
- Изменения схемы БД, auth/security/infra вне спецификации.
- Удаления файлов, `git reset`, разрушительные действия.
- Подключение новых зависимостей без обсуждения.
- Изменение конфигурации проекта (.env, CI/CD, Dockerfile).
- Любые действия с внешними сервисами (API-вызовы, деплой).

## 4) Протокол отклонений (deviation rules)

Если при выполнении задачи нужно отклониться от плана:

**Минорное отклонение** (дополнительный файл, мелкая правка соседнего модуля):
- Допустимо, если не меняет поведение вне scope задачи.
- Зафиксировать в SUMMARY.md как "deviation".

**Существенное отклонение** (новая зависимость, изменение архитектуры, > 3 файлов):
- **Остановиться** и запросить подтверждение.
- Не продолжать до получения ответа.

## 5) Лимиты итерации

- Одна задача из PLAN.md за итерацию.
- Максимум 2 запуска тяжёлых тестов на задачу.
- Если тесты падают 2 раза подряд по **разным** причинам — **остановиться** и спросить.

## 6) Stop-policy (когда остановиться)

Остановись и задай вопросы, если:
- Не хватает требований (неясны инварианты, edge cases).
- Есть риск для данных или безопасности.
- Задача выходит за scope плана.
- Нужна установка пакетов или сетевое действие.
- Обнаружен баг вне scope → зафиксировать в `docs/bugs.md`, не чинить.
- Аутентификация/авторизация CLI-инструмента не удалась.

## 7) Протокол коммитов

Каждая выполненная задача коммитится немедленно:

```
<type>(<phase>-<plan>): <краткое описание>
```

Типы: `feat`, `fix`, `test`, `refactor`, `perf`, `chore`, `docs`.

Примеры:
```
feat(03-02): добавить endpoint регистрации
test(03-02): добавить тесты аутентификации
fix(03-02): исправить валидацию email
```

## 8) Формат результата (каждая задача)

После выполнения задачи агент предоставляет:
1. Что сделано (краткое описание).
2. Файлы изменены (список).
3. Команды проверки (что запустить).
4. Результат проверок (pass/fail).
5. Риски/края (что могло сломаться).
6. Коммит (хэш и сообщение).

## 9) Безопасность

- Никаких секретов в коде, логах, коммитах.
- Не ослаблять авторизацию/роли "для теста".
- Валидация входных данных на границе системы.
- Не коммитить `.env`, credentials, ключи API.
- При обнаружении уязвимости — зафиксировать, не игнорировать.
