# Инициализация проекта: PROJECT.md

PROJECT.md — центральный артефакт проекта. Это "живой" документ,
который обновляется по мере развития проекта.

---

## Что содержит PROJECT.md

| Секция | Назначение |
|--------|------------|
| **What This Is** | Текущее описание продукта (2-3 предложения) |
| **Core Value** | ОДНА вещь, которая важнее всего. Если всё остальное сломается — это должно работать |
| **Requirements** | Validated / Active / Out of Scope |
| **Context** | Фон, техническое окружение, предыдущий опыт |
| **Constraints** | Жёсткие ограничения: стек, сроки, бюджет, совместимость |
| **Key Decisions** | Решения, которые ограничивают будущую работу |

## Принципы

### Core Value — якорь приоритизации
Одно предложение, которое определяет всё. Когда возникают трейдоффы,
Core Value решает, что важнее.

**Хорошо:** "Пользователи могут делиться контентом с единомышленниками."
**Плохо:** "Хороший пользовательский опыт."

### Requirements — гипотезы до подтверждения
- **Validated** — отправлено и подтверждено ценностью.
- **Active** — текущий scope, строим к этому.
- **Out of Scope** — явные исключения с обоснованием.

Active requirements — это гипотезы. Они становятся Validated только после
отправки и подтверждения.

### Constraints — всегда с обоснованием
Ограничение без обоснования будет оспорено. Всегда указывай "почему".

```
- **Tech stack**: Python + FastAPI — команда имеет экспертизу
- **Timeline**: 2 недели — демо инвесторам
- **Budget**: Бесплатные тиры — пока нет финансирования
```

### Key Decisions — аудит решений
Значимые выборы, которые влияют на будущую работу:

| Решение | Обоснование | Результат |
|---------|-------------|-----------|
| SQLite вместо Postgres | MVP, один пользователь | — Pending |
| HTMX вместо React | Простота, серверный рендеринг | ✓ Good |

## Эволюция PROJECT.md

### После каждой фазы:
1. Требования опровергнуты? → Переместить в Out of Scope.
2. Требования подтверждены? → Переместить в Validated.
3. Появились новые? → Добавить в Active.
4. Решения? → Добавить в Key Decisions.
5. "What This Is" ещё актуально? → Обновить если нет.

### После каждого милестоуна:
1. Полный обзор всех секций.
2. Core Value — всё ещё верный приоритет?
3. Out of Scope — причины ещё актуальны?
4. Обновить Context текущим состоянием.

## Для существующих проектов (brownfield)

1. Сначала проанализируй кодовую базу (структура, стек, конвенции).
2. **Infer Validated requirements** из существующего кода.
3. **Gather Active requirements** от пользователя — что строим дальше.
4. Инициализируй PROJECT.md:
   - Validated = выведено из кода.
   - Active = цели пользователя.
   - Out of Scope = границы от пользователя.
   - Context = включает текущее состояние кодовой базы.
