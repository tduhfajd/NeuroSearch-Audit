# Принципы Phase Forge: контекстная инженерия и фазовая разработка

## 1) Контекстная инженерия — основа качества

ИИ выдаёт отличный результат, **если** дать ему правильный контекст.
Большинство людей этого не делают.

Phase Forge решает это через структурированные артефакты:

| Артефакт | Назначение |
|----------|------------|
| `PROJECT.md` | Видение проекта, всегда загружается первым |
| `REQUIREMENTS.md` | Скоупированные требования v1/v2 с трассировкой по фазам |
| `ROADMAP.md` | Куда идём, что сделано |
| `STATE.md` | Решения, блокеры, позиция — память между сессиями |
| `CONTEXT.md` | Решения по реализации конкретной фазы |
| `PLAN.md` | Атомарные задачи с верификацией |
| `SUMMARY.md` | Что произошло, что изменилось — зафиксировано в истории |

ИИ видит не "весь репозиторий", а **контракты, решения и границы**.

## 2) Деградация контекста (context rot) — главный враг

По мере заполнения контекстного окна качество падает:
- Генерация становится менее точной.
- Агент начинает "экономить" и упрощать.
- Накапливаются ошибки предыдущих шагов.

**Решение:** каждый план выполняется в **свежем контексте**.
Подагент получает чистое окно (~200K токенов) исключительно для реализации.
Никакого "накопленного мусора".

## 3) Фаза — единица прогресса

Проект разбивается на **фазы** (5–12 штук для стандартного проекта).
Каждая фаза:
- доставляет что-то целостное,
- имеет чёткие критерии успеха (наблюдаемое поведение),
- содержит 2–3 плана (атомарных задания),
- проходит полный цикл: discuss → plan → execute → verify.

Фаза НЕ равна PR или задаче. Фаза = "кирпич" функциональности.

## 4) Волновое выполнение (wave execution)

Планы внутри фазы группируются в **волны** по зависимостям:

```
Волна 1 (параллельно)      Волна 2 (параллельно)      Волна 3
┌──────────┐ ┌──────────┐  ┌──────────┐ ┌──────────┐  ┌──────────┐
│ План 01  │ │ План 02  │→ │ План 03  │ │ План 04  │→ │ План 05  │
│ Модель   │ │ API      │  │ Интерф.  │ │ Тесты    │  │ Интегр.  │
└──────────┘ └──────────┘  └──────────┘ └──────────┘  └──────────┘
```

- Независимые планы → одна волна → параллельное выполнение.
- Зависимые планы → следующая волна → ждут завершения зависимостей.
- Конфликты файлов → один план или последовательные волны.

## 5) Мульти-агентная оркестрация

Каждый этап использует паттерн: тонкий **оркестратор** порождает специализированных агентов.

| Этап | Оркестратор | Агенты |
|------|-------------|--------|
| Исследование | Координирует, представляет результаты | 4 параллельных исследователя: стек, фичи, архитектура, подводные камни |
| Планирование | Валидирует, управляет итерациями | Планировщик создаёт планы, проверщик верифицирует, цикл до успеха |
| Выполнение | Группирует в волны, отслеживает прогресс | Исполнители реализуют параллельно, каждый со свежим контекстом |
| Верификация | Представляет результаты, маршрутизирует дальше | Верификатор проверяет код vs цели, дебаггеры диагностируют сбои |

Оркестратор **никогда** не делает тяжёлую работу. Он порождает агентов, ждёт, интегрирует результаты.

## 6) Спек-ориентированная разработка (spec-driven)

Каждая задача в плане — это **спецификация**, а не описание:

```xml
<task type="auto">
  <name>Создать endpoint аутентификации</name>
  <files>src/api/auth/login.py</files>
  <action>
    Использовать jose для JWT (не PyJWT — проблемы с ESM).
    Валидировать credentials по таблице users.
    Возвращать httpOnly cookie при успехе.
  </action>
  <verify>curl -X POST localhost:8000/api/auth/login → 200 + Set-Cookie</verify>
  <done>Валидные credentials → cookie, невалидные → 401</done>
</task>
```

Точные инструкции. Никаких догадок. Верификация встроена.

## 7) Верификация на каждом уровне

Три уровня проверки:

1. **Автоматическая** — тесты, линтер, типизация после каждой задачи.
2. **Goal-backward** — код доставляет то, что обещала фаза (не просто "задачи выполнены").
3. **UAT (User Acceptance Testing)** — человек проверяет, что фича реально работает.

Это не "тесты ради тестов", а **контракт качества**: задача считается выполненной,
только когда все три уровня пройдены.

## 8) Атомарные коммиты

Каждая задача → свой коммит. Не "один коммит на фазу", а гранулярная история:

```
abc123f docs(03-02): завершить план регистрации
def456g feat(03-02): добавить email подтверждение
hij789k feat(03-02): реализовать хэширование паролей
lmn012o feat(03-02): создать endpoint регистрации
```

Преимущества:
- `git bisect` находит точную задачу со сбоем.
- Каждая задача откатывается независимо.
- Чистая история для ИИ в будущих сессиях.

## 9) Адаптивность без блокировки

- Добавь фазу к текущему милестоуну.
- Вставь срочную работу между фазами (десятичная нумерация).
- Заверши милестоун и начни новый.
- Скорректируй планы без пересборки всего.

Система адаптируется. Ты никогда не заблокирован.

## 10) Рабочие режимы

### Режим A: Полный цикл (production)
Все этапы: discuss → plan → execute → verify.
Research + plan-check + verifier включены.

### Режим B: Быстрый (прототип)
Пропуск research, plan-check, verifier.
Для знакомых доменов и быстрых итераций.

### Режим C: Quick mode (ad-hoc)
Для багфиксов, мелких фич, конфиг-изменений.
Planificator + executor, без полной фазовой обвязки.
