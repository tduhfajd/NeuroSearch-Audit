# Верификация: автотесты, goal-backward, UAT

Верификация — не опция, а обязательный этап. Три уровня проверки
гарантируют, что фаза доставила то, что обещала.

---

## Уровень 1: Автоматическая верификация

### После каждой задачи
- Запустить `<verify>` из задачи.
- Линтер/типизация должны быть чистыми.
- Если verify падает → починить → повторить (max 2 раза).

### После каждой волны
- Запустить полный набор тестов.
- Все тесты зелёные → следующая волна.
- Есть падения → диагностика и фикс.

### Проверка стабов
Существование файла ≠ реализация. Четыре уровня:

| Уровень | Описание | Пример |
|---------|----------|--------|
| Exists | Файл создан | `user.py` существует |
| Substantive | Содержит реальный код (не стаб) | Нет `pass`, `TODO`, `NotImplementedError` |
| Wired | Подключён к системе | Роут зарегистрирован, модель импортирована |
| Functional | Работает end-to-end | API отвечает корректно |

Типичные паттерны стабов (флаги для поиска):
- `pass` в теле функции
- `TODO`, `FIXME`, `PLACEHOLDER`
- `NotImplementedError`, `raise NotImplementedError`
- Хардкожа вместо реальной логики
- Пустые обработчики ошибок

## Уровень 2: Goal-backward верификация

Проверка **от целей фазы назад к коду**.
Не "задачи выполнены", а "цели достигнуты".

### Три измерения

**1. Truths (Наблюдаемые истины):**
Из Success Criteria фазы. Каждый критерий проверяется фактически.

```markdown
| Truth | Verified | Evidence |
|-------|----------|----------|
| Пользователь может создать профиль | ✅ | POST /api/profiles → 201 |
| Профиль показывает аватар | ✅ | GET /api/profiles/1 → avatar_url |
| Невалидный bio → ошибка | ✅ | POST с bio > 500 → 422 |
```

**2. Artifacts (Артефакты):**
Ожидаемые файлы существуют и содержат реальную реализацию (не стабы).

**3. Wiring (Связи):**
Компоненты подключены друг к другу:
- Компонент → API (фронтенд вызывает бэкенд)
- API → Database (роут использует модель)
- Form → Handler (форма отправляет данные)
- State → Render (состояние отображается)

### Результат: VERIFICATION.md

```markdown
## Observable Truths
- ✅ 5/5 критериев подтверждены

## Required Artifacts
- ✅ 8/8 файлов существуют и содержат реальный код

## Key Wiring
- ✅ Все связи работают

## Gaps
(нет)

## Verdict: PASS
```

Если есть gaps → план исправлений.

## Уровень 3: UAT (User Acceptance Testing)

Ручная проверка человеком. Автоматическая верификация проверяет код,
UAT проверяет **пользовательский опыт**.

### Процесс

1. **Извлечь тестируемые поведения** из SUMMARY.md файлов фазы.
2. **Провести по одному:** "Можешь залогиниться?" → да/нет/описание проблемы.
3. **При проблемах:** автоматическая диагностика (debug agent).
4. **Результат:** UAT.md с результатами и планом исправлений.

### Формат UAT.md

```markdown
## Tests

### 1. Создание профиля
expected: Форма отправляется, профиль появляется в списке
result: pass

### 2. Загрузка аватара
expected: Изображение загружается, отображается в профиле
result: issue
reported: "Загружается, но показывает broken image"
severity: major

## Summary
total: 6
passed: 5
issues: 1

## Gaps
- truth: "Аватар отображается в профиле"
  status: failed
  severity: major
  reason: "broken image после загрузки"
  root_cause: ""  # Заполняется после диагностики
```

### Определение severity

Severity **выводится** из описания, не спрашивается:

| Описание | Severity |
|----------|----------|
| Падение, ошибка, неработает совсем | blocker |
| Не работает, ничего не происходит, неправильное поведение | major |
| Работает, но... медленно, странно, мелкая проблема | minor |
| Цвет, шрифт, отступ, выравнивание | cosmetic |

По умолчанию: **major** (безопасный дефолт).

### После UAT

**Все тесты прошли:** → переход к следующей фазе.
**Есть проблемы:**
1. Запусти диагностику (debug agents параллельно).
2. Получи root causes.
3. Создай план исправлений.
4. Выполни исправления.
5. Re-verify.

## Стратегия валидации (Nyquist Layer)

Для каждой фазы создаётся **карта верификации** до начала реализации:

| Task | Requirement | Test Type | Command | Exists |
|------|-------------|-----------|---------|--------|
| 03-01-01 | AUTH-01 | unit | `pytest tests/test_auth.py::test_signup -x` | ✅ |
| 03-01-02 | AUTH-02 | integration | `pytest tests/test_auth.py::test_verify -x` | ❌ W0 |

- **W0 (Wave 0):** тестовые файлы/скаффолды, которые создаются **до** реализации.
- **Каждая задача** должна иметь автоматическую верификацию.
- **Максимум 3 задачи подряд** без автоматической верификации.
