# Выполнение: волновое исполнение планов

Выполнение фазы — координированный процесс: оркестратор управляет,
исполнители реализуют в свежих контекстах.

---

## Принцип свежих контекстов

Каждый план выполняется в **изолированном** контекстном окне.
Исполнитель получает:
- PROJECT.md (видение).
- PLAN.md (задачи).
- STATE.md (позиция и решения).
- CONTEXT.md (решения по фазе, если есть).
- RESEARCH.md (исследование, если есть).

Он **не** получает мусор предыдущих планов, обсуждений, ошибок.
Результат: ~200K токенов чисто на реализацию.

---

## Волновое выполнение

### Алгоритм

1. Прочитать все PLAN.md файлы фазы.
2. Распарсить frontmatter: `wave`, `depends_on`.
3. Сгруппировать в волны.
4. Для каждой волны:
   a. Запустить все планы волны **параллельно**.
   b. Дождаться завершения всех.
   c. Проверить результаты.
   d. Перейти к следующей волне.

### Обработка ошибок

- **План упал:** зафиксировать ошибку в SUMMARY.md, продолжить с другими планами волны.
- **Зависимость упала:** зависимые планы пропускаются с пометкой "blocked".
- **2 падения подряд:** остановиться, запросить направление у человека.

## Протокол выполнения задачи

Для каждой задачи внутри плана:

### 1. Выполнение
- Прочитать `<action>` — что нужно сделать.
- Реализовать минимальный diff.
- Не выходить за scope задачи.

### 2. Верификация
- Запустить команду из `<verify>`.
- Если падает — починить и запустить снова.
- Максимум 2 попытки. Если не проходит → checkpoint.

### 3. Коммит
```
<type>(<phase>-<plan>): <описание задачи>
```

Одна задача = один коммит. Не группировать.

### 4. Переход
- Следующая задача из того же плана.
- Или завершение плана → SUMMARY.md.

---

## Протокол отклонений

### Минорное (допустимо)
- Дополнительный файл, не указанный в `files_modified`.
- Мелкая правка соседнего модуля для совместимости.
- Зафиксировать в SUMMARY.md:
  ```
  deviation: "Добавлен utils/validators.py — не указан в плане, нужен для валидации"
  ```

### Существенное (требует подтверждения)
- Новая зависимость.
- Изменение архитектурного решения.
- Более 3 файлов вне плана.
- **Остановиться**, описать проблему, ждать решения.

## User Setup задачи

Некоторые задачи требуют действий человека (создание API ключей,
настройка сервисов). Такие задачи помечаются `type="human-action"`:

```xml
<task type="human-action">
  <name>Настроить Stripe API ключ</name>
  <action>
    1. Создать аккаунт на stripe.com
    2. Получить test API key
    3. Добавить в .env: STRIPE_API_KEY=sk_test_...
  </action>
  <verify>python -c "import stripe; stripe.api_key='...'; print('OK')"</verify>
  <done>Stripe API ключ доступен в .env</done>
</task>
```

Автоматизируй где возможно (CLI-команды, скрипты), но не делай
сетевых запросов или операций с секретами без подтверждения.
