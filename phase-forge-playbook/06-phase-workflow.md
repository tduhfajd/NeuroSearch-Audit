# Фазовый цикл: discuss → plan → execute → verify

Каждая фаза проходит четыре этапа. Пропуск этапов допустим
для быстрых итераций, но снижает качество.

---

## Обзор цикла

```
┌──────────────────────────────────────────────────┐
│                  ДЛЯ КАЖДОЙ ФАЗЫ:                │
│                                                  │
│  ┌────────────────────┐                          │
│  │   DISCUSS          │  ← Зафиксировать         │
│  │   Обсуждение       │    предпочтения           │
│  └──────────┬─────────┘                          │
│             │                                    │
│  ┌──────────▼─────────┐                          │
│  │   PLAN             │  ← Исследование +        │
│  │   Планирование     │    Планы + Проверка      │
│  └──────────┬─────────┘                          │
│             │                                    │
│  ┌──────────▼─────────┐                          │
│  │   EXECUTE          │  ← Волновое выполнение   │
│  │   Выполнение       │    со свежими контекстами│
```

## 1) DISCUSS — обсуждение реализации

**Цель:** захватить решения, которые нужны нижестоящим агентам.

**Что происходит:**
1. Анализируется фаза → выявляются "серые зоны" (где могут быть разные подходы).
2. Ты выбираешь, что обсудить (не обязательно всё).
3. Для каждой области — глубокое обсуждение до ясности.

**Какие "серые зоны" выявляются:**
- **Визуальные фичи** → layout, плотность, взаимодействия, пустые состояния.
- **API/CLI** → формат ответа, флаги, обработка ошибок, verbose.
- **Контент** → структура, тон, глубина, поток.
- **Организация** → критерии группировки, именование, дубликаты.

**Результат:** `CONTEXT.md` — документ решений для фазы.

**Кто потребляет:**
- **Исследователь** → фокусирует исследование (нашли "card layout" → исследуем библиотеки карточек).
- **Планировщик** → знает что зафиксировано ("infinite scroll" → задача включает виртуализацию).

**Чем глубже обсуждение — тем точнее результат.** Пропустишь — получишь "разумные дефолты".

---

## 2) PLAN — планирование фазы

**Цель:** создать атомарные планы выполнения с задачами и верификацией.

**Что происходит:**
1. **Research** (опционально) — исследование домена: стек, паттерны, подводные камни.
2. **Plan** — создание 2–3 планов (PLAN.md) с задачами в XML-формате.
3. **Verify** — проверка планов vs requirements. Цикл до 3 итераций.

**Координация агентов:**
```
plan-phase
    │
    ├── Исследователь (4 параллельных)
    │     ├── Стек
    │     ├── Фичи
    │     ├── Архитектура
    │     └── Подводные камни
    │           │
    │     ┌─────▼──────┐
    │     │ RESEARCH.md │
    │     └─────┬──────┘
    │           │
    │     ┌─────▼──────┐
    │     │ Планировщик│ ← Читает PROJECT, REQUIREMENTS,
    │     │            │   CONTEXT, RESEARCH
    │     └─────┬──────┘
    │           │
    │     ┌─────▼────────┐   ┌────────┐
    │     │ Проверщик    │──>│ PASS?  │
    │     └──────────────┘   └───┬────┘
    │                            │
    │                       Да   │  Нет → loop (до 3x)
    │                        │   │
    │                  ┌─────▼──────┐
    │                  │ PLAN files │
    │                  └────────────┘
    └── Done
```

**Результат:** `RESEARCH.md` + один или несколько `PLAN.md`.

---

## 3) EXECUTE — выполнение фазы

**Цель:** реализовать все планы фазы с атомарными коммитами.

**Что происходит:**
1. Анализ зависимостей → группировка в волны.
2. Волна 1: независимые планы выполняются параллельно (свежие контексты).
3. Волна 2: зависимые от волны 1 планы.
4. ... до завершения всех планов.
5. Post-execution верификация: код vs цели фазы.

**Ключевое правило:** каждый исполнитель получает **свежее** контекстное окно.
Никакого накопленного мусора. 200K токенов чисто на реализацию.

**Протокол задачи:**
1. Выполнить задачу.
2. Запустить верификацию (`<verify>` из задачи).
3. Починить если нужно.
4. Коммит: `<type>(<phase>-<plan>): <описание>`.
5. Следующая задача.

**Результат:** `SUMMARY.md` для каждого плана + `VERIFICATION.md` для фазы.

---

## 4) VERIFY — верификация фазы

**Цель:** убедиться, что фаза доставила то, что обещала.

**Три уровня:**

### Автоматическая верификация
- Тесты проходят.
- Линтер/типизация чистые.
- Артефакты существуют (не стабы, а реальная реализация).

### Goal-backward верификация
- Проверка от **целей фазы назад к коду** (не "задачи выполнены", а "цели достигнуты").
- Три измерения: truths (поведение), artifacts (файлы), wiring (связи).

### UAT (User Acceptance Testing)
- Ручная проверка: человек тестирует фичи по списку.
- "Можешь залогиниться?" → Да/Нет/описание проблемы.
- Проблемы → автоматическая диагностика → план исправлений.

**Если всё пройдено:** переход к следующей фазе.
**Если есть проблемы:** diagnose → fix-plan → re-execute → re-verify.

---

## Между фазами

1. **Обнови STATE.md** — позиция, прогресс, решения.
2. **Обнови ROADMAP.md** — отметь фазу как Complete.
3. **Очисти контекст** (`/clear` в Claude Code).
4. **Начни следующую фазу** с discuss.

Чистка контекста **критична**. GSD спроектирован вокруг свежих контекстов.
Если качество падает — очисти и используй resume-work для восстановления.
