# Use Cases

## UC-001 Быстрый пресейл-аудит
- Акторы: Account Manager, SEO Specialist.
- Триггер: поступила заявка на аудит сайта.
- Предусловия: есть URL сайта, задан ruleset Runet-first.
- Основной сценарий:
1. Менеджер запускает аудит.
2. Система выполняет crawl + extraction + scoring.
3. Система формирует audit package и промты.
4. Внешний AI генерирует клиентский отчет и КП.
5. Менеджер презентует пакет клиенту.
- Результат: клиент получает доказательный отчет и предложение работ.
- Acceptance: время цикла не превышает внутренний SLA агентства.

## UC-002 Техническая подготовка работ
- Акторы: Tech Lead, SEO Specialist.
- Триггер: клиент согласовал работы.
- Предусловия: audit package доступен.
- Основной сценарий:
1. Команда открывает технический отчет.
2. Проблемы конвертируются в backlog.
3. Для задач назначаются приоритеты P0/P1/P2.
4. Формируется этапный план внедрения.
- Результат: команда получает исполнимый план работ с критериями приемки.
- Acceptance: каждая задача привязана к источнику доказательства.

## UC-003 Контроль качества обновленного сайта
- Акторы: SEO Specialist, QA Analyst.
- Триггер: завершен этап внедрения.
- Предусловия: есть baseline-результаты предыдущего аудита.
- Основной сценарий:
1. Запускается повторный аудит.
2. Система сравнивает coverage/contradictions/scores с baseline.
3. Формируется дельта-отчет по улучшениям и остаточным рискам.
- Результат: подтверждение результата и список доработок.
- Acceptance: изменения выражены в измеримых метриках ECS/CS/ARS.

## UC-004 Массовая обработка лидов агентства
- Акторы: Owner/CEO, Presale Team.
- Триггер: пакет входящих лидов за неделю/месяц.
- Предусловия: очередь заявок и лимиты ресурса.
- Основной сценарий:
1. Запускаются аудиты по списку доменов.
2. Система автоматически собирает пакеты артефактов.
3. Команда ранжирует лиды по LVI и сложности.
4. Для приоритетных лидов генерируются индивидуальные КП.
- Результат: стандартизированная фабрика аудитов.
- Acceptance: для каждого лида есть одинаковая структура пакета.

## UC-005 Работа с конфликтными данными
- Акторы: Analyst, Tech Lead.
- Триггер: обнаружены contradictions high severity.
- Предусловия: доступны источники конфликтов.
- Основной сценарий:
1. Аналитик проверяет конфликтные URL.
2. Назначается владелец исправления.
3. Формируется remediation task с критериями.
4. После правки запускается точечная переоценка.
- Результат: устранены критичные несоответствия.
- Acceptance: high severity contradiction снижен или закрыт с доказательством.
