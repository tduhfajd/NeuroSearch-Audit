# PRD

## Product Statement
NeuroSearch должна автоматически превращать анализ сайта в структурированную факт-базу и набор документов для продаж и реализации работ по SEO/GEO/AEO/AIO/LEO.

## Functional Requirements

### FR-001 Crawl Orchestration
- ID: `FR-001`
- Название: Управление обходом сайта
- Описание: Система принимает домен/URL и запускает управляемый обход с очередью URL, лимитами и политиками глубины.
- Причина/ценность: Без полного обхода невозможно достоверно оценить структуру и проблемы сайта.
- Входные данные: стартовый URL, лимиты обхода, robots/sitemap настройки.
- Выходные данные: список посещенных URL, crawl status, ошибки обхода.
- Acceptance Criteria:
1. Для каждого URL фиксируется статус получения (успех/ошибка/пропуск).
2. Очередь обхода предотвращает повторный обход уже посещенных страниц.
3. Результат обхода сохраняется в структурированном виде.
- Ограничения: В MVP допускается ограничение размера сайта и глубины обхода.

### FR-002 Multi-layer Extraction
- ID: `FR-002`
- Название: Извлечение технических и смысловых данных
- Описание: Система извлекает HTML/metadata/links и нормализованные смысловые блоки страницы.
- Причина/ценность: Нужна единая база для технического и маркетингового анализа.
- Входные данные: HTML/DOM/рендер-данные страницы.
- Выходные данные: `page_blocks.json`, мета-атрибуты, структура ссылок.
- Acceptance Criteria:
1. Для каждой страницы определяется `page_type`.
2. Для поддерживаемых типов блоков выставляются `presence` и `confidence`.
3. Формат выхода соответствует контракту данных.
- Ограничения: В MVP-1 поддерживается ограниченный набор блоков.

### FR-003 Entity Graph
- ID: `FR-003`
- Название: Построение графа сущностей
- Описание: Система выделяет сущности и связи между ними, включая связь сущности с URL-источником.
- Причина/ценность: Позволяет оценивать полноту и противоречивость на уровне смыслов.
- Входные данные: `page_blocks`, структурные признаки страниц.
- Выходные данные: `entities.json`, `entity_graph.json`.
- Acceptance Criteria:
1. Каждая сущность имеет тип, label и список источников.
2. Каждая связь имеет тип relation и source evidence.
3. Граф может быть использован в coverage/contradiction анализе.
- Ограничения: В MVP-1 базовые типы сущностей (org/service/geo/price/document/faq).

### FR-004 Coverage Analysis
- ID: `FR-004`
- Название: Оценка полноты контента
- Описание: Система проверяет обязательные блоки по типам страниц/сущностей и формирует gaps.
- Причина/ценность: Даёт измеримый объем работ и приоритизацию.
- Входные данные: `page_blocks`, `entities`, ruleset coverage.
- Выходные данные: `coverage_report.json`.
- Acceptance Criteria:
1. Для каждой сущности есть coverage score 0..1.
2. Для каждого gap указаны impact и priority.
3. Отчет отделяет missing vs partial coverage.
- Ограничения: Ruleset MVP ограничен основными шаблонами страниц.

### FR-005 Contradiction Detection
- ID: `FR-005`
- Название: Выявление противоречий
- Описание: Система обнаруживает конфликтующие факты (цены, сроки, условия, география, контакты).
- Причина/ценность: Противоречия критичны для доверия, AIO и конверсии.
- Входные данные: extracted facts по URL и сущностям.
- Выходные данные: `contradictions.json` с severity/risk.
- Acceptance Criteria:
1. Конфликт содержит минимум 2 источника URL.
2. У конфликта есть severity (high/medium/low).
3. У конфликта есть business/AI risk tags.
- Ограничения: В MVP-1 фиксируются основные классы конфликтов.

### FR-006 Scoring Engine
- ID: `FR-006`
- Название: Расчет оценок готовности
- Описание: Система рассчитывает ECS, CS, ARS и LVI.
- Причина/ценность: Нужны KPI для приоритизации и КП.
- Входные данные: coverage, contradictions, technical metrics.
- Выходные данные: `ai_readiness_scores.json` и score summary.
- Acceptance Criteria:
1. Для каждой страницы доступен набор SEO/GEO/AEO/AIO/LEO score.
2. Для каждого score документирована формула или правило.
3. Итоговые баллы воспроизводимы на одинаковом наборе входов.
- Ограничения: Формулы в MVP допускают rule-based модель без ML.

### FR-007 Recommendation Engine
- ID: `FR-007`
- Название: Нормализованные рекомендации
- Описание: Система превращает findings в actionable рекомендации с приоритетом и критериями приемки.
- Причина/ценность: Нужен прямой переход к выполнению работ.
- Входные данные: scores + findings + rules.
- Выходные данные: `recommendations.json`.
- Acceptance Criteria:
1. Рекомендация привязана к проблеме и URL.
2. Есть priority, effort и expected impact.
3. Есть формулировка acceptance criteria.
- Ограничения: В MVP-1 используется шаблонная библиотека рекомендаций.

### FR-008 Prompt Pack Generation
- ID: `FR-008`
- Название: Генерация промтов для внешнего AI
- Описание: Система готовит набор промтов под 5 документов с ссылкой на JSON-факты.
- Причина/ценность: Независимость от конкретной LLM и быстрый выпуск документов.
- Входные данные: JSON-артефакты анализа.
- Выходные данные: `prompts/*.md`.
- Acceptance Criteria:
1. Каждый промт содержит роль, аудиторию, входные файлы, структуру документа.
2. Каждый промт содержит запрет на выдумывание данных.
3. Каждый промт требует ссылок на доказательства.
- Ограничения: В MVP-1 шаблоны промтов на русском языке.

### FR-009 Audit Package Assembly
- ID: `FR-009`
- Название: Сборка audit package
- Описание: Система упаковывает все JSON и промты в единый deliverable для передачи AI-агенту.
- Причина/ценность: Ускоряет handoff и исключает потерю данных.
- Входные данные: outputs модулей FR-001..FR-008.
- Выходные данные: каталог `audit_package/<audit_id>/`.
- Acceptance Criteria:
1. Пакет содержит полный набор обязательных файлов.
2. Есть manifest с версиями правил и timestamp.
3. Пакет валидируется перед выдачей.
- Ограничения: Формат пакета фиксируется версией schema.

### FR-010 Evidence-first Reporting Policy
- ID: `FR-010`
- Название: Политика доказательности
- Описание: Любой вывод должен ссылаться на URL/факт/score или явно помечаться как data gap.
- Причина/ценность: Снижение галлюцинаций и повышение доверия.
- Входные данные: findings и источники.
- Выходные данные: evidence tags в отчетах/промтах.
- Acceptance Criteria:
1. Для каждого finding есть ссылка на источник или маркер missing data.
2. Генерация документа блокируется при отсутствии источников для high severity claims.
3. Проверка evidence coverage входит в quality gate.
- Ограничения: Полная автоматическая проверка текстов AI-выхода может потребовать Post-MVP tooling.

## Facts
- В `idea.md` зафиксированы Smart Crawler, semantic crawl, entity graph, coverage/contradictions, scoring и пакет документов.

## Assumptions
- PRD описывает целевое состояние MVP-1 + MVP-2, а не только текущую реализацию.

## Hypotheses
- Evidence-first политика повысит конверсию коммерческих документов за счет доверия.

## Open Questions
- Какой минимальный набор доменов/страниц обязателен для корректного скоринга.
- Какую часть scoring нужно калибровать на реальных кейсах агентства до запуска.

## Traceability
- `idea.md`: фрагменты про pipeline crawl->analysis->docs, выходные JSON, scoring ECS/CS/ARS/LVI, требования anti-hallucination и пакет из 5 документов.
